# Flow Trigger: "Charging state changed from X to Y"

## Goal

Add a new flow trigger card that fires when the EV charger's charging state
transitions from a specific value to another specific value. This complements
the existing standard trigger on `evcharger_charging_state` (which only allows
filtering on the new value), enabling more precise flow logic such as
"only when going from Paused to Charging".

This is **additive**: no existing trigger card is modified or replaced.

## Trigger card

- **ID**: `charging_state_changed_from_to`
- **Title**:
  - EN: "Charging state changed from ... to ..."
  - NL: "Oplaadstatus is veranderd van ... naar ..."
- **Args**: two dropdowns, `from` and `to`, each with the same five values
  as the standard `evcharger_charging_state` capability:
  - `plugged_in_charging` ‚Äî Charging / Opladen
  - `plugged_in_discharging` ‚Äî Discharging / Ontladen
  - `plugged_in_paused` ‚Äî Paused / Gepauzeerd
  - `plugged_in` ‚Äî Plugged in / Aangesloten
  - `plugged_out` ‚Äî Unplugged / Niet aangesloten
- **Tokens**: `from` (string) and `to` (string), so downstream flow steps can
  reference both old and new values.
- **Device filter**: `driver_id=smartevse` (matches existing trigger cards).

`plugged_in_discharging` is included in the dropdown for parity with the
standard Homey card. SmartEVSE does not support V2G/V2H, so this app never
emits that value in practice ‚Äî the option is present but will not fire.

## Behaviour

The trigger fires when the device's derived `evcharger_charging_state`
capability transitions to a different value. It uses the same value computed
by `deriveChargingState()` in `lib/derived.ts`; no new derivation logic is
introduced.

Rules:

- **First state after init**: do not fire (no previous value known ‚Äî avoids
  a spurious trigger on app startup or device pairing).
- **No actual change**: do not fire (the derived value can be recomputed on
  every MQTT message; only real transitions count).
- **Real change**: fire with `from = previous`, `to = new`. Homey's
  `runListener` filters on the args so only flows whose `from` and `to` both
  match the actual transition will run.

## Implementation

### `.homeycompose/flow/triggers/charging_state_changed_from_to.json` (new)

Standard Homey trigger card definition with the dropdown args and tokens
described above.

### `drivers/smartevse/device.ts`

- Add a private field `lastChargingState: ChargingState | undefined`.
- In `updateDerived()`, after computing the new charging state `cs`:
  - If `this.lastChargingState !== undefined && this.lastChargingState !== cs`,
    call `tryTrigger('charging_state_changed_from_to', { from: this.lastChargingState, to: cs })`.
  - Then set `this.lastChargingState = cs`.

This keeps the change-detection in one place ‚Äî the function that already
owns the derived value.

### `drivers/smartevse/driver.ts`

- Add a `registerFlowTriggers()` method, called from `onInit()` alongside
  `registerFlowActions()` and `registerFlowConditions()`.
- Inside, register a `runListener` on `charging_state_changed_from_to` that
  returns `args.from === tokens.from && args.to === tokens.to`.

### `app.json`

Auto-regenerated by homeycompose from the new trigger JSON file. Not edited
by hand.

## Testing

Add a unit test for the change-detection logic. Two reasonable options:

1. Extract the comparison into a small pure helper in `lib/derived.ts`
   (e.g. `detectChargingStateChange(prev, next)` returning the transition or
   `null`) and unit-test it in `test/derived.test.ts`.
2. Test the trigger wiring at the device level via a lightweight harness.

Option 1 is preferred: keeps the test fast and pure, and the helper is
trivial to call from `device.ts`.

Test cases:

- `prev = undefined`, `next = 'plugged_in'` ‚Üí no transition.
- `prev = 'plugged_in'`, `next = 'plugged_in'` ‚Üí no transition.
- `prev = 'plugged_in_paused'`, `next = 'plugged_in_charging'` ‚Üí
  transition `{from: 'plugged_in_paused', to: 'plugged_in_charging'}`.

## Files touched

- `.homeycompose/flow/triggers/charging_state_changed_from_to.json` ‚Äî new
- `drivers/smartevse/device.ts` ‚Äî track previous state, fire trigger
- `drivers/smartevse/driver.ts` ‚Äî register `runListener` for arg filtering
- `lib/derived.ts` ‚Äî small `detectChargingStateChange` helper (if option 1)
- `test/derived.test.ts` ‚Äî tests for the helper
- `app.json` ‚Äî regenerated by homeycompose

## Out of scope

- No changes to the existing `charger_state_changed` trigger (raw
  `charger_state` string).
- No changes to `deriveChargingState()` itself.
- No new "from any" / "to any" wildcard option ‚Äî both args are required
  concrete values, since the standard Homey card already covers the
  "to any specific value" case.

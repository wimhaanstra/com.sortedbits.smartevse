# Charging-state From-To Trigger Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a new flow trigger card "Charging state changed from X to Y" that fires when the device's derived `evcharger_charging_state` transitions between two specific values, complementing (not replacing) the existing standard trigger.

**Architecture:** Pure helper `detectChargingStateChange(prev, next)` lives in `lib/derived.ts` and is unit-tested. The device tracks the previous derived state and calls the helper inside `updateDerived()`, firing a new device trigger card with `from`/`to` tokens on real transitions. The driver registers a `runListener` on the new card that filters by `args.from` and `args.to` so only matching flows run.

**Tech Stack:** TypeScript, Homey Apps SDK v3, Jest, homeycompose for flow card definitions.

- [ ] **Step 3: Implement the helper**

```typescript
export interface ChargingStateChange {
  from: ChargingState;
  to: ChargingState;
}

export function detectChargingStateChange(
  prev: ChargingState | undefined,
  next: ChargingState,
): ChargingStateChange | null {
  if (prev === undefined) return null;
  if (prev === next) return null;
  return { from: prev, to: next };
}
```

The existing helper always passes `{}` as state, which would make our arg-filter comparison always false. Allow callers to pass a `state` payload while keeping existing call sites unchanged.

In `drivers/smartevse/device.ts`, replace the entire `tryTrigger` method:

```typescript
    const card = api.getDeviceTriggerCard(id);

  }
```

with:


## Notes

- `app.json` is regenerated by homeycompose at app run/publish time — do not hand-edit it. None of the tasks above touch it.
- `plugged_in_discharging` is included in the dropdown for parity with the standard Homey card. SmartEVSE does not produce this value via `deriveChargingState`, so the new trigger will not fire transitions to/from it in practice — that is intentional.
- All logging in this app must go through `this.log` / `this.error`. The implementation here uses the existing `tryTrigger` helper which already handles errors silently; no new logging is introduced.
